## 1. 项目名称、主题、面向用户与使用场景

- **项目名称**：AI日程转换助手（AISchedule）。
- **主题**：智能日程管理 App——把一句口语化表达，经 LLM 解析为结构化日程草稿，用户确认后写入本地日历，并在到点前提醒。
- **面向用户**：大学生（课程/作业/社团/小组会议）、上班族（会议/截止日期）、健忘型用户。
- **使用场景**：聊天或临时想到一件事，直接说一句话即可生成可编辑、可提醒的日程，免去传统待办 App 逐项填表。

## 2. 主要功能模块（6 个核心功能）

1. **一句话创建日程**：输入口语 → AI 自动提取标题/时间/地点/提醒。
2. **AI 澄清与纠错**：时间含糊时返回 `needClarification` 与追问。
3. **日历/列表视图**：CalendarView 选日期 + RecyclerView 展示当天日程。
4. **手动填写（降级）**：无 API Key/断网时仍可手动新建与编辑。
5. **提醒通知**：AlarmManager 到点 Notification，点击进详情。
6. **API Key 与隐私设置**：provider/baseUrl/model 可配，Key 本地加密保存。

## 4. 数据库设计（Room 三表）

- **EventRecord（events）**：`id`(主键)、`title`、`startTime`/`endTime`(epoch 毫秒)、`location`、`reminderMinutes`、`sourceText`、`status`。
- **ParseHistory（parse_history）**：`id`、`inputText`、`modelName`、`jsonResult`、`confidence`、`createdAt`、`isApplied`。
- **ApiConfig（api_config）**：`id`(固定 1)、`provider`、`baseUrl`、`modelName`、`keyAlias`、`createdAt`、`updatedAt`。
- **DAO**：`EventDao`/`HistoryDao`/`ApiConfigDao` 提供 `@Insert/@Update/@Delete/@Query`，读返回 `LiveData`，另有 `getByIdSync`、`getAllCursor/getByIdCursor`(供 ContentProvider)。
- **AppDatabase**：`@Database(version=1, exportSchema=false)` 单例；`ScheduleRepository` 作为唯一数据入口，写操作在后台线程、读返回 LiveData。

## 5. 关键技术实现（对应实现方式说明）

- **RecyclerView**：看板页 `EventAdapter`(ListAdapter+DiffUtil) 展示 标题/时间/地点/提醒（4 字段）；历史页 `HistoryAdapter` 展示 原句/JSON/可信度。
- **Room + CRUD**：新增(确认页)、查询(看板/详情)、修改、删除 全实现；写操作经 `AppExecutors.diskIO()` 后台执行。
- **数据从输入页保存到数据库**：首页原句 →(可选)AI 解析回填 → 确认页校验 → `Repository.insertEventSync()` 后台写入。
- **库数据显示到 RecyclerView**：看板页经 `ContentResolver.query()` 读取 events 表 Cursor → 映射 EventRecord → 列表。
- **修改/删除后界面刷新**：Repository 写入后 `notifyChange(CONTENT_URI_EVENTS)`；看板页 `ContentObserver` 自动重新查询刷新（详情页用 LiveData observe）。
- **ContentProvider 经 URI 访问**：`ScheduleProvider` 暴露 `content://com.qiu.aischedule.provider/events` 与 `/events/{id}`，实现 query/insert/update/delete 全 4 项 + getType；看板页通过 ContentResolver 读取（已用 adb content 验证）。
- **网络访问**：`LlmClient` 用 OkHttp `enqueue`（不在主线程）调用 DeepSeek `/chat/completions`（OpenAI 兼容），Gson 解析 `choices[0].message.content`，截取 JSON 容错，回填确认页；失败降级手动填写。
- **Notification**：`ReminderScheduler` 用 `AlarmManager.setAndAllowWhileIdle` 在 `startTime-reminderMinutes` 定时；`AlarmReceiver` 到点 → `Notifier`（渠道 IMPORTANCE_HIGH）发通知，点击 `PendingIntent` 跳详情；保存设提醒、更新重设、删除取消；菜单"测试通知(5秒)"演示整链路。

## 6. 开发中遇到的问题与解决（≥3）

1. **主线程访问 Room 崩溃**：Repository 所有写操作改到后台线程（`AppExecutors.diskIO()`）；同步读方法注明须在后台调用。
2. **`Calendar.set` 7 参数编译错误**：选日处误用 7 参数重载——改为逐字段 `set` 并单独置 `MILLISECOND=0`，保证"当天 0 点"毫秒归零。
3. **API 26+ 通知不显示**：创建 `NotificationChannel`（minSdk 24 用 `SDK_INT>=O` 兼容）；Android 13+ 运行时请求 `POST_NOTIFICATIONS`。
4. **LLM 偶尔返回非纯 JSON**：截取首个 `{...}` 再 Gson 解析，字段用 `optStr/optInt/optBool` 容错；失败降级手动填写。
5. **`am broadcast` 无法触达非导出 Receiver**（验证阶段）：属系统安全限制；临时导出验证通知链路成功后还原 `exported=false`。

## 7. 项目总结（≥300 字）

本项目「AI日程转换助手」以"一句话创建日程"为切入点，把大语言模型的自然语言理解能力与 Android 本地数据管理结合起来，完成了一个可运行、可演示的完整 App。技术上覆盖了课程要求的全部要点：用 Room（Entity/DAO/Database）做本地存储并实现增查改删；用两个 RecyclerView 分别展示日程与解析历史；用 ContentProvider 以 URI 方式对外暴露日程数据并作为看板页的读取路径，配合 ContentObserver 实现"修改/删除后界面自动刷新"；用 OkHttp 异步调用 DeepSeek 接口并用 Gson 解析、回填字段，严格不在主线程；用 AlarmManager + Notification 实现到点提醒并处理了 Android 13+ 的运行时权限。

开发中我坚持"先有最小可运行版本、再逐步加功能"的思路：阶段 2 完成时 App 已经是一个脱离 AI 也能用的普通日程管理器，之后才接入网络、通知与 ContentProvider，从而保证了任何时候都能演示、都不会因为网络或 Key 缺失而崩溃。AI 辅助主要用于生成样板代码（实体、Adapter、通知示例等），我对每段生成代码都核对了包名、控件 ID、Manifest 注册、Gradle 依赖与数据库字段的一致性，并按自己理解改写（如 Room 用 public 字段、换日改用内存过滤、Receiver 读库放后台线程等），所有提示词与修改过程均记录在 `docs/prompts.md`。整体上，这个项目让我更系统地理解了 Android 的分层架构、异步与生命周期，以及如何把"AI 能力"稳妥地嵌入到一个传统移动应用里。

## 已知取舍 / 未解决问题（如实说明）

- 提醒使用 `setAndAllowWhileIdle`（非精确闹钟），未启用 `SCHEDULE_EXACT_ALARM` 权限；Doze 下可能有轻微延迟。如需精确提醒可后续加该权限并改 `setExactAndAllowWhileIdle`。
- ContentProvider 与测试 Receiver 出于演示需要 `exported`，生产环境应改为不导出或加权限校验。
- AI 解析依赖用户自备 DeepSeek API Key（在设置页填入，本地加密存储）。
