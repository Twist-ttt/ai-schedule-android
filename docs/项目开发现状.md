## 一、项目定位与一句话概括

**「AI日程转换助手」（AISchedule）** 是一款把「自然语言一句话」自动转化为「结构化本地日程 + 到点提醒」的 Android 原生 App。核心闭环：

```
用户输入一句话  →  LLM 解析为结构化草稿  →  确认页人工核对/修改  →  写入本地 Room 日历  →  到点 AlarmManager + Notification 提醒
```

设计上的两条「保命线」：
1. **可降级**：无 API Key / 断网 / LLM 失败时，自动退化为普通「手动填写」日程管理器，**任何时刻都能演示、不会因外部依赖崩溃**。
2. **数据自洽**：时间全程以 epoch 毫秒（long）流转，字符串仅出现在 UI 边界；写入一律后台线程，读返回 LiveData 自动刷新。

### 4.2 数据流图（读 / 写两条路径）

```
┌──────────────────────────────────────────────────────────────┐
│                            UI 层                              │
│  MainActivity  AiConfirmActivity  ScheduleListActivity  ...   │
└───────────┬───────────────────────────────┬───────────────────┘
            │ 读（LiveData observe）          │ 写（insert/update/delete）
            ▼                               ▼
┌──────────────────────┐         ┌───────────────────────────┐
│  ScheduleRepository  │ ◄─────► │  AppExecutors.diskIO()     │
│  （单例·唯一入口）    │         │  （单线程后台）             │
└──────┬───────┬───────┘         └───────────────────────────┘
       │       │ 写完调
       │       │ ContentResolver.notifyChange(CONTENT_URI)
       │       ▼
       │  ┌─────────────────┐    触发看板页 ContentObserver
       │  │ ScheduleProvider │ ◄──── 看板页 query() 读 Cursor
       │  └─────────────────┘
       ▼
┌──────────────────────────────────────────────┐
│  Room: EventDao / HistoryDao / ApiConfigDao   │
│  → ai_schedule.db (events / parse_history /   │
│                    api_config)                │
└──────────────────────────────────────────────┘
```

**关键约束（在代码中 enforced）**：
- **UI 只与 `ScheduleRepository` 交互**，不直接碰 DAO / Room / ContentProvider。
- **读返回 `LiveData`**，UI `observe` 后数据变更自动刷新。
- **写一律经 `AppExecutors.diskIO()`（单线程）切后台**，事件写完调 `notifyChange(CONTENT_URI_EVENTS)` 触发看板页 `ContentObserver` 刷新。
- **同步读/写**（`getEventSync` / `insertEventSync` 等）**必须在后台线程调用**，供通知、详情预取、网络回调使用。

### 4.3 线程模型

`AppExecutors` 只有两把线程池：

| 线程 | 实现 | 用途 |
|---|---|---|
| `diskIO()` | `Executors.newSingleThreadExecutor()` | 单线程，所有 Room 读写、Cursor 查询、网络请求前置读配置 |
| `mainThread()` | `Handler(post 到 MainLooper)` | 回调切回主线程刷新 UI |

**铁律**：永远不在主线程访问 Room（会抛 `IllegalStateException`）。回调用自定义 `ScheduleRepository.Callback<T>`，**刻意不引入 `java.util.function`**（作业保持依赖最小化）。

## 五、各模块设计详解

### 5.1 数据层（Room 三表）

所有时间字段统一存 **epoch 毫秒（long）**，字符串只出现在 UI 边界（日期/时间选择器格式化）。`DateUtils.combine(...)` 合成 millis，**不把时间当字符串解析**。

| 表 | 实体 | 关键字段 | 索引 |
|---|---|---|---|
| `events` | `EventRecord` | `id`(自增) · `title` · `startTime` · `endTime`(long) · `location` · `reminderMinutes` · `sourceText` · `status` | `startTime` |
| `parse_history` | `ParseHistory` | `id` · `inputText` · `modelName` · `jsonResult` · `confidence`(float) · `createdAt` · `isApplied` | `createdAt` |
| `api_config` | `ApiConfig` | `id`(**固定=1** 单行) · `provider` · `baseUrl` · `modelName` · `keyAlias` · `createdAt` · `updatedAt` | — |

`AppDatabase`：`@Database(version=1, exportSchema=false)` 单例（`ai_schedule.db`）。

**DAO 设计要点**：
- 读返回 `LiveData`（`getAll` / `observeById` / `getByDayRange`）。
- 同步读 `getByIdSync` 供 ContentProvider / 通知在后台线程使用。
- `EventDao` 另有 `getAllCursor` / `getByIdCursor`（返回 `Cursor`）**供 ContentProvider 直接对外暴露**——CP 不做对象映射，直接吐 Room 的 Cursor。

### 5.2 ScheduleRepository（数据入口）

单例，双重检查锁。它把「线程切换 + 通知 ContentObserver」封装在内部，UI 调用时无需关心线程：

```java
public void insertEvent(EventRecord event) {
    executors.diskIO().execute(() -> {
        eventDao.insert(event);
        notifyEventsChanged();   // → ContentResolver.notifyChange(CONTENT_URI_EVENTS)
    });
}
```

`insertEventSync` / `insertHistorySync` 返回 id，供确认页拿到 id 后立即设置 AlarmManager 提醒、供网络回调在后台线程写留痕。

### 5.3 ContentProvider（`ScheduleProvider`）

暴露 `events` 表，**实现 query / insert / update / delete 全 4 项 + `getType`**：

| URI | 匹配码 | 操作 |
|---|---|---|
| `content://com.qiu.aischedule.provider/events` | `CODE_EVENTS_DIR` | query(全表) / insert |
| `content://com.qiu.aischedule.provider/events/{id}` | `CODE_EVENTS_ITEM` | query / update / delete |

- `UriMatcher` 区分列表/单条；`query` 返回的 Cursor 调 `setNotificationUri`，使观察者能收到变更。
- **看板页 `ScheduleListActivity` 经 `ContentResolver.query()` + `ContentObserver` 读取**——直接在 UI 体现 CP 用法。
- Manifest `exported=true`（便于 `adb content` 验证与视频演示）；**CP 方法运行在调用方线程**，调用方需自行切后台。

> **诚实说明**：CP 的 `insert/update/delete` 已实现并通过 adb 验证（插入→查到→删除→清空），但 App 自身 UI 的写操作走的是 `Repository`（内部调 DAO），并未经 `ContentResolver` 调 CP。CP 在本项目中主要承担「看板页读路径」+「对外暴露能力演示」双重角色——这避免了 UI 写路径绕一圈 ContentResolver 的无谓开销，同时仍完整覆盖了课程对 CP 的考核点。

### 5.4 AI 网络（`LlmClient`）

单例，OkHttp 异步 + Gson，调用 **OpenAI 兼容的 `/chat/completions`**（默认 DeepSeek：`baseUrl=https://api.deepseek.com`，`model=deepseek-v4-flash`，设置页可改）。

**关键设计**：
- 整个 `parse()` 流程（读配置 → 读 Key → 发请求 → 解析 → 写历史 → 回调）**全在 `diskIO()` 后台线程**，**绝不在主线程**。
- **System Prompt** 要求只输出 JSON；解析时**截取首个 `{...}`** 容错（兼容模型偶尔包裹代码块/多余文字）。
- 字段类型不稳用 `optStr/optInt/optBool` 逐个容错（如 `reminderMinutes` 可能是数字也可能是字符串）。
- 解析结果写入 `ParseHistory`（**AI 辅助过程留痕**，对应评分项「AI 提示词记录」），再主线程回调 UI 回填确认页。
- API Key 经 `SecretStore`（EncryptedSharedPreferences + Keystore）加密读取，**明文 Key 不入库不入日志**。
- 无 Key / 网络失败时回调 `onError`，UI 弹 Toast **降级为手动填写**。

### 5.5 通知提醒链路

```
确认页保存  ──►  ReminderScheduler.schedule(eventId, startTime - reminderMinutes)
                         │  AlarmManager.setAndAllowWhileIdle(RTC_WAKEUP)
                         │  requestCode = eventId（每条日程独立、可单独取消）
                         ▼  到点
                 AlarmReceiver (exported=false)
                         │  后台线程读库
                         ▼
                 Notifier.showEventReminder()
                         │  建渠道 schedule_reminder (IMPORTANCE_HIGH)
                         │  PendingIntent 跳详情页
                         ▼
                 NotificationManagerCompat.notify()
```

- 用 `setAndAllowWhileIdle` **免申请 `SCHEDULE_EXACT_ALARM` 权限**，Doze 下也能唤醒。
- 保存设提醒、**更新重设**、**删除取消**（`ReminderScheduler.cancel`，按 eventId 精确取消）。
- `MainActivity` 首次进入请求 `POST_NOTIFICATIONS`（Android 13+ 运行时权限）。
- 菜单「测试通知(5秒)」用 `eventId=-1` 演示整链路，通知点击跳首页。

#### `BaseActivity` —— edge-to-edge 根治方案（本项目最值得讲的工程迭代）

`targetSdk = 36` 强制 edge-to-edge，导致**装饰层 ActionBar 与内容重叠**（半透明 ActionBar 时标题盖在内容上）。前几轮用 `statusBar` / `actionBarSize` padding 打补丁**全部失败**。最终定型解法（配合 `NoActionBar` 主题 + 各布局内 `MaterialToolbar`）：

1. `setDecorFitsSystemWindows(false)` 拥抱 edge-to-edge；
2. 把布局里的 `MaterialToolbar`（id=`toolbar`）`setSupportActionBar`——标题/菜单/返回由此承载；
3. 把**系统栏 inset 作为内容根的 padding**——Toolbar（首子视图）落在状态栏下方、内容落在 Toolbar 下方。**标题与内容同在内容层、纵向堆叠，重叠在结构上不可能发生。**

子页面只需重写 `showUpNavigation()` 返回 true 即显示左上返回箭头。`attachBaseContext` 强制 `Locale.SIMPLIFIED_CHINESE`，统一界面与系统组件（日期/时间选择器）语言。

#### 视觉风格：Glassmorphism（毛玻璃）
- 背景：渐变 + 柔光斑；卡片/输入框/按钮用半透明 + 模糊霜层 + 投影高光分层。
- 日期/时间选择器：**自研玻璃转轮 Dialog**（`GlassDatePickerDialog` 快捷选月 / `GlassTimePickerDialog` 时分转轮），与整体风格统一，而非默认 Material 选择器。

#### 数据刷新
- 看板页：`ContentObserver` + `ListAdapter + DiffUtil`，写后自动重新查询并差异刷新。
- 详情页：`LiveData observe`，数据变更自动回填。

## 六、关键技术难点与解决（≥5 项，均有据可查）

| # | 难点 | 根因 | 解决 | 代码位置 |
|---|---|---|---|---|
| 1 | 主线程访问 Room 崩溃 | Room 默认禁止主线程 IO | Repository 所有写包进 `diskIO()`；同步读注明须后台调用 | `ScheduleRepository` |
| 2 | `Calendar.set` 7 参数重载编译错误 | 误用不存在的方法签名 | 改逐字段 `set` 并单独置 `MILLISECOND=0`，保证"当天 0 点"归零 | `DateUtils.dayRange` / `combine` |
| 3 | MaterialDatePicker 跨时区日期偏移 | `selection` 是 UTC 零点，本地时区下偏一天 | `utcMidnightToLocal` / `localMidnightToUtc` 显式换算 | `DateUtils:71-92` |
| 4 | API 26+ 通知不显示 | 缺 NotificationChannel | `ensureChannel`（`SDK_INT>=O` 兼容）+ Android 13+ 运行时请求 `POST_NOTIFICATIONS` | `Notifier` / `MainActivity` |
| 5 | LLM 偶尔返回非纯 JSON | 模型可能包裹代码块/多余文字 | 截取首个 `{...}` 再 Gson 解析；字段 `optStr/optInt/optBool` 容错；失败降级手动填写 | `LlmClient.parseSchedule` |
| 6 | edge-to-edge 下标题与内容重叠 | targetSdk 36 强制 edge-to-edge，装饰层 ActionBar 叠在内容上 | 内容层 MaterialToolbar + inset padding（见 5.6） | `BaseActivity` |
| 7 | `am broadcast` 无法触达非导出 Receiver | 系统安全限制 | 临时导出验证通知链路成功后**还原 `exported=false`** | `AlarmReceiver` / Manifest |

> 第 6 项经历了 **6 轮迭代**（见 git 历史「阶段 6」：从「关闭 edge-to-edge」→「实色状态栏」→「paddingTop 打补丁」→「取消装饰层 ActionBar，标题入内容层 MaterialToolbar」最终根治）。这是本项目工程严谨度的代表案例。

## 八、AI 辅助开发过程与质量管控

本项目深度使用 AI 辅助，但坚持「**AI 生成样板、人工核对与改写**」的原则：

1. **提示词全程记录**：`docs/prompts.md` 按轮次记录每次 AI 提示词与我的修改；应用内 `ParseHistory` 表 additionally 留存每次「原句 → AI 返回 JSON」的过程（含模型名、可信度、是否被应用）。
2. **逐段核对一致性**：对 AI 生成的每段代码，核对 **包名、XML 控件 ID、Manifest 注册、Gradle 依赖、Room 字段** 五项全局一致。
3. **按理解改写**：例如 Room 实体改用 public 字段、换日改用内存过滤、Receiver 读库放后台线程、自定义 Callback 不依赖 `java.util.function`、ContentProvider 直接吐 Cursor 等。
4. **敏捷提交**：每个增量 `git commit`（Conventional Commits）+ push 到 GitHub，并同步更新 `docs/CHANGELOG.md`（三件套：① 做了什么 ② AI 提示词与我的修改 ③ 问题与解决）。

## 九、已知取舍与待办（如实说明）

### 已知取舍（设计层面的有意选择）
1. **提醒用 `setAndAllowWhileIdle`（非精确闹钟）**：未启用 `SCHEDULE_EXACT_ALARM` 权限，Doze 下可能有轻微延迟。如需精确提醒可后续加该权限并改 `setExactAndAllowWhileIdle`。
2. **ContentProvider / MainActivity 为 `exported=true`**：出于演示与 `adb` 验证需要；生产环境应改为不导出或加权限校验。
3. **AI 解析依赖用户自备 DeepSeek API Key**：在设置页填入，本地加密存储；无 Key 时降级手动填写。
4. **`endTime` 当前恒等于 `startTime`**：UI 未提供结束时间选择，保留字段以备扩展（如时长/跨天）。
5. **CP 的 insert/update/delete 未被 UI 调用**：UI 写走 Repository，CP 主要作看板读路径 + 能力演示（见 5.3 诚实说明）。
