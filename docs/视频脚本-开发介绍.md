### ⏱️ 1:25–2:55　功能模块与主要页面（命中"功能完整度 / ≥3 页面"）

🖥️ **画面**：可在 AS 里依次点开各 Activity 文件名简单带过；不必运行

🎙️ **口播**：
> 应用一共有七个页面，六个核心页面加一个多匹配候选页。
>
> 第一个是**首页输入页**，用户在这里输入一句话，点"AI 解析"，应用就会调用大模型理解意图。
>
> 第二个是**确认页**，它有三种模式复用同一套界面：AI 解析后预填、手动从零填写、以及编辑已有日程。
>
> 第三个是**看板页**，用日历加列表展示所有日程，这里我用了 RecyclerView，每条卡片展示时间、标题、地点、提醒四个字段；数据是通过 ContentProvider 来读取的；顶部还有一个今日摘要。

> 第四个是**详情页**，可以查看、修改、删除单条日程。

> 第五个是**解析历史页**，这是项目里的第二个 RecyclerView，它记录每一次"用户原句 → AI 返回 JSON"的过程，对应作业里"AI 提示词使用记录"这一项。

### ⏱️ 2:55–4:15　关键技术实现（命中 Room / 网络 / CP / Notification 等技术分）

🖥️ **画面**：依次打开对应源码文件（ScheduleRepository → LlmClient → ScheduleProvider → Notifier/ReminderScheduler）

🎙️ **口播**：
> 关键技术我挑几个重点讲。
>
> **数据库**用的是 Room，三张表。一个设计细节是：所有时间字段我都统一存成毫秒时间戳的 long，不存字符串，这样就避免了时区错乱和字符串解析的麻烦。读写都走 Repository，写操作一律切到后台线程——因为 Room 不允许在主线程访问，会直接抛异常。
>
> **列表**用 RecyclerView 的 ListAdapter 加 DiffUtil，数据变化时是差异刷新，不是整列表重刷，性能更好。
>
> **网络**这块用 OkHttp 的异步请求，天然不在主线程，再用 Gson 解析返回的 JSON。我特别做了几层容错：大模型偶尔会在 JSON 外面加代码块或说明文字，我会先截取第一个花括号括起来的部分再解析；每个字段也用容错的方式读取，防止类型不稳导致崩溃。如果没有 API Key 或者网络失败，应用会自动降级成手动填写，保证任何时候都能正常演示。
>
> **通知**用 AlarmManager 在日程开始前定时，到点弹通知，点击能跳到对应日程的详情页。也做了通知渠道，以及 Android 13 以上的运行时权限申请。
>
> **ContentProvider** 实现了增、删、改、查全部四项操作，看板页的列表数据就是通过它来读的。

### ⏱️ 5:10–5:50　踩坑与解决（至少一个，原件硬性要求）

🖥️ **画面**：打开 `app/src/main/java/com/qiu/aischedule/ui/BaseActivity.java`，按这四块指：
- `BaseActivity.java:20-34`（类注释——根因 + 三步解法，可直接念，最有说服力）
- `BaseActivity.java:51`（① `setDecorFitsSystemWindows(false)` 拥抱 edge-to-edge）
- `BaseActivity.java:61-65`（② 找到布局里的 `MaterialToolbar` 并 `setSupportActionBar`，标题进入内容层）
- `BaseActivity.java:73-78`（③ 系统栏 inset 作为内容根 padding——标题与内容同层纵向堆叠，重叠结构上不可能）

🎙️ **口播**：
> 最后讲一个印象最深的坑。
>
> 我把 targetSdk 设成了 36，它会强制开启 edge-to-edge，结果页面标题栏和内容重叠在一起——状态栏的时间、电量会盖在标题上面。这个问题我前前后后改了差不多六七轮：先是给状态栏设实色，再是加各种内边距 padding，但都只是治标，换个页面又冒出来。
>
> 最后我才意识到，根因是"装饰层的 ActionBar 叠在内容之上"这种**结构问题**，光调偏移量是解决不了的。于是我换了个思路：主题里取消 ActionBar、每个页面改用放在内容层里的 MaterialToolbar、再把系统栏的高度作为内容区的内边距。这样一来，标题和内容处在同一个层里、纵向排列，重叠在结构上就不可能发生了。
