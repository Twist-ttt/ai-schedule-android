# AI日程转换助手 · 软件设计评审报告

> 评审对象：AI日程转换助手（AISchedule）当前版本
> 评审视角：**纯软件设计**——产品概念、信息架构、任务流、交互、视觉语言、内容、可访问性、情感化
> 评审日期：2026-06-21
> 说明：本报告不涉及任何技术实现（存储、网络、通知机制等），仅从「用户看到的、用到的」层面评估。

---

## 一、评审方法与维度

本评审以**真实任务流走查**为主，辅以界面静态审阅，从八个设计维度逐一评判：

1. 产品概念与价值主张
2. 信息架构与导航
3. 核心任务流
4. 交互设计（反馈、状态、错误、破坏性操作）
5. 视觉设计语言
6. 内容设计（文案、语气、术语）
7. 可访问性与包容性
8. 情感化设计

每个发现标注**严重度**：🔴 P0（损害核心体验或致数据损失）｜🟠 P1（明显体验折损）｜🟡 P2（打磨项）。

## 二、产品概念与价值主张

**概念**：一句话口语 → 结构化日程 → 到点提醒。把「自然语言理解」作为日程创建的主入口，区别于传统待办 App 的逐项填表。

**评价**：概念清晰、切口锋利。"一句话"是一个能被一句话讲清的差异点，这是产品定位上的优势。首页把「AI 解析」作为视觉与交互的第一主角，价值主张传达得很直接。

**值得肯定的设计哲学**：

- **永不死胡同**：AI 失败 / 无 Key / 断网时，始终保留「手动填写」这条退路。这是产品韧性的体现——一个会"坏掉"的 AI 功能比没有更糟，而这里把 AI 设计成"锦上添花、非必需"，是成熟的产品判断。
- **AI 与人的职责分工**：AI 出草稿、人做确认。没有让 AI 直接写库，也没有让用户从零填空。这条"草稿→确认"的中庸路线，既享受了 AI 的省力，又守住了用户对最终结果的掌控。

**概念层面的隐忧**：

- 🟠 **P1**：产品的差异点是"AI 创建"，但一个日程工具被打开 10 次，可能有 8 次是为了**看**日程、2 次才是**建**日程。当前应用把"创建"放在每次启动的首屏，而把"回顾"放在二级页。**创建被高估、回顾被低估**——这是概念落地后的优先级倒挂。（详见第三节）

## 三、信息架构与导航

### 3.1 结构

```
首页（输入页）
  ├─ [AI 解析]    → 确认页（AI 模式）→ 看板页
  ├─ [手动填写]   → 确认页（手动模式）→ 看板页
  ├─ [我的日程]   → 看板页 →（点条目）→ 详情页
  └─ 溢出菜单     → 历史 / 设置 / 测试通知

看板页（日程基地）
  ├─ FAB [+]      → 确认页（手动模式）
  ├─ 点条目       → 详情页（改 / 删）
  └─ 溢出菜单     → 历史 / 设置 / 测试通知
```

导航层级浅（最深 3 层），回退路径清晰，这是**优点**。

### 3.2 问题

- 🟠 **P1 · 启动落点**：每次打开应用都落在"输入页"。但对**回头用户**而言，最高频的需求是"今天/接下来有什么"。建议把启动落点改为**看板页**（带今日摘要），把"新建"收到 FAB——这样回顾与创建都触手可及，且符合大多数日程类工具的心智模型（先看后建）。输入页可保留为 FAB 或一个显眼入口。
- 🟡 **P2 · 入口冗余**：「我的日程」按钮（首页）与「保存后自动跳转看板」、以及看板 FAB，构成了三条通往"创建/看板"的路径。冗余本身不致命，但首页三个并列大按钮（AI 解析 / 手动填写 / 我的日程）信息层级偏平，**主次没有拉开**。建议：AI 解析为唯一主按钮（实色、最大），其余两个降为文字链或次要按钮。
- 🟡 **P2 · 缺少"即将到来"视角**：看板以日历选日为主，缺少"未来 7 天""下一项"这类前瞻视图。今日摘要部分缓解了"今天"的前瞻，但跨天的全局节奏感缺失。

## 四、核心任务流评审

### 4.1 主流程：一句话 → 日程

```
输入一句话 → [AI 解析] → (等待) → 确认页回填 → 校对/改 → [保存] → 反馈弹窗 → 看板
```

**优点**：

- 等待期有进度指示（进度条 + 按钮禁用），未让用户对着静止界面干等。
- 保存后有**明确的反馈对话框**（"将在开始前 N 分钟提醒你 / 未设置提醒 / 提醒时间已过"），给了用户"提醒已就绪"的确定感——这是很好的闭环设计。
- 确认页字段全部可改，AI 回填只是"建议"而非"判决"，用户始终握有最终编辑权。

**问题**：

- 🔴 **P0 · 澄清问题转瞬即逝**：当 AI 判断信息不足（如时间含糊），会返回一个追问（"needClarification"）。但当前实现是——**弹一个 Toast 显示追问，然后照样跳进确认页**。Toast 几秒后消失，用户进入确认页时已经看不到 AI 在问什么，只能面对一堆空字段发呆。**这是主流程上的一处断裂**：AI 的引导没有延续到它该起作用的现场。
  - **建议**：把追问以**常驻提示条**的形式放在确认页顶部（紧贴原始输入下方），直到用户填齐对应字段。让 AI 的"问"和用户的"答"发生在同一屏。

- 🟠 **P1 · AI 失败的降级是"被动"的**：解析失败时只弹 Toast"AI 解析失败…（可改用手动填写）"。用户读完 Toast，仍停在首页，需要自己再去点「手动填写」，并把刚才输入的那句话**重新带过去**（当前手动模式不带原文）。降级路径存在，但摩擦明显。
  - **建议**：失败 Toast 改为**内联的失败卡片 + 「用手动填写继续」按钮**，点击后带着已输入的原文直接进确认页手动模式。让降级"一键延续"而非"从头再来"。

- 🟡 **P2 · 等待期无取消**：AI 解析期间无法中止请求，网络慢时用户被钉在等待态。

### 4.2 管理流程：看 → 改 → 删

- 🔴 **P0 · 删除无二次确认**：详情页点「删除」**立即执行**，仅补一个"已删除"Toast，随即返回。日程是用户亲手创建的数据，**一次误点即永久丢失，且无撤销**。这是本应用最严重的设计缺陷。
  - **建议**：删除前弹确认对话框（"删除「标题」？此操作不可撤销"）；或采用"软删除 + 撤销条"（删除后底部出现「撤销」5 秒）。二者皆可，但不能是现在的"点即删"。

- 🟠 **P1 · 没有"完成"概念**：日程一旦创建就永远是"已保存"，UI 上无法标记某项已完成/已过。一个日程工具不能让用户勾掉已完成的事项，**核心生命周期缺失一环**。已过期或已完成的日程和待办混在一起，列表会逐渐被"僵尸条目"填满。
  - **建议**：详情页/卡片支持「标记完成」；已完成项视觉弱化（灰阶/删除线）并可筛选隐藏。

- 🟠 **P1 · 缺少时长/结束时间**：所有日程都是"一个时间点"，没有"持续多久"。但现实中"下午 3 点开会，2 小时"是基本语义——AI 其实能解析出时长，只是产品没有承接。这限制了"日程冲突感知""一天排满了吗"等高价值能力。

## 五、交互设计

### 5.1 反馈与状态

| 场景 | 现状 | 评价 |
|---|---|---|
| 保存成功 | 对话框三态反馈 | ✅ 优秀（确定感强） |
| 更新成功 | Toast | 🟡 可，但弱于新建（不对称） |
| 删除 | Toast，无确认 | 🔴 见 4.2 |
| AI 解析中 | 进度条 + 按钮禁用 | ✅ |
| AI 澄清 | Toast（瞬时） | 🔴 见 4.1 |
| AI 失败 | Toast（被动） | 🟠 见 4.1 |
| 权限请求 | 冷启动直接弹系统框 | 🟠 见下 |

- 🟠 **P1 · 通知权限冷启动直弹**：首次进入即在 onCreate 请求通知权限，**没有任何前置的价值说明**。用户面对一个刚打开的陌生 App 弹出的系统权限框，默认倾向拒绝；一旦拒绝，后续"到点提醒"这一核心卖点就废了。**权限请求时机和理由都没设计**。
  - **建议**：推迟到**用户第一次创建带提醒的日程、即将保存时**再请求，并先用应用内文案解释"为了到点提醒你，需要通知权限"。情境化、动机明确的请求，通过率显著更高。

### 5.2 破坏性操作

- 🔴 删除：无确认、无撤销（同 4.2，此处再次强调其严重性）。
- 🟡 **P2 · 设置页保存即覆盖**：API 配置保存是直接覆盖单行配置，无"放弃修改/还原"。对低频设置可接受。

### 5.3 容错与边界

- 🟡 **P2 · 空态不可操作**：看板"当天暂无日程"、历史"暂无解析记录"都是纯文字，没有引导下一步动作。好的空态应是"行动号召"——如"当天暂无日程，[+] 新建一个"。
- 🟡 **P2 · 提醒模型单一**：只有"提前 N 分钟"一种提醒，且每个日程仅一个。缺少"到点提醒""多个提醒（如前一天 + 当天）""重复日程"。当前模型能演示链路，但作为日常工具偏弱。

## 六、视觉设计语言

### 6.1 风格定位

采用 **Glassmorphism（毛玻璃）**：渐变背景 + 彩色柔光斑 + 半透明霜化卡片 + 高光描边。这是一套**有辨识度、有现代感**的视觉语言，在课程作业中足够出挑。

### 6.2 优点

- ✅ **风格贯彻彻底**：从 Toolbar、卡片、输入框、按钮到自研的日期/时间转轮选择器，毛玻璃语言贯穿每一个界面，没有"默认 Material 选择器"破坏整体感的不和谐。这种**端到端的视觉一致性**是高完成度的表现。
- ✅ **设计系统已成型**：圆角、间距、尺寸统一收口为设计令牌（卡片 20dp / 输入 16dp / 按钮 24dp / 页边距 20dp 等），不再"每个卡片不一样"。这是从"调 UI"升级到"有设计系统意识"的关键一步。
- ✅ **信息层级清晰**：日程卡片以"时间左侧大字锚点 + 标题 + 地点·提醒副行"组织，扫读时第一眼抓到"几点"，符合日程类信息的真实阅读优先级。
- ✅ **昼夜双适配**：色彩有夜间版本，照顾了不同环境下的可读性。

### 6.3 问题

- 🟠 **P1 · 半透明带来的对比度风险**：毛玻璃依赖半透明叠加，文字落在"透出渐变与光斑"的背景上，**在某些光斑位置文字对比度可能不足**（尤其次要文字 `glass_text_secondary`）。建议对所有文本/背景组合做对比度核查（WCAG AA，正文 ≥ 4.5:1），必要时给卡片底色提高不透明度或给文字加微弱阴影兜底。
- 🟡 **P2 · 强调色双轨**：同时使用靛蓝（标签/聚焦/时间锚点）与青色（提醒/今日摘要副行）两种强调色。二者功能边界需更明确，否则用户难以建立"什么颜色=什么含义"的稳定预期。
- 🟡 **P2 · 日历控件未玻璃化**：日历是系统原生控件，与周围毛玻璃面板在质感上有落差，是整页风格里唯一的"外来物"。

## 七、内容设计（文案、语气、术语）

### 7.1 语气

整体语气**平实、克制、偏功能**。没有多余的卖萌，符合"工具"定位。首页副标题"把一句话，变成可提醒的日程"是一句合格的价值主张。

### 7.2 问题

- 🟠 **P1 · 历史页向用户暴露原始 JSON**：解析历史页直接展示 AI 返回的 **JSON 文本**给用户。JSON 是开发者的语言，不是用户的语言。普通用户看到 `{"title":"...","reminderMinutes":15}` 会困惑。虽然这服务于"展示 AI 过程"的作业意图，但从设计角度，**面向用户的信息应以人话呈现**（如把 JSON 渲染成"标题/时间/地点/提醒"的字段卡），原始 JSON 可折叠或作为"高级/调试"视图。
- 🟡 **P2 · 设置页文案偏技术**：provider / baseUrl / model / API Key 直接以技术术语示人，且 baseUrl 字段需要用户自己拼。对一个"自带 Key 的极客用户"够用，但对更广的用户群门槛偏高。可考虑提供"DeepSeek / OpenAI / 其它"的预设选择，baseUrl 等随之自动填充。
- 🟡 **P2 · 测试通知留在生产菜单**：溢出菜单的"测试通知（5秒后）"是开发/演示工具，出现在面向用户的菜单里是噪声。建议在发布版本中移除或藏到"关于/调试"页。
- ✅ **做得好的细节**：保存反馈的三态文案（"将在开始前 N 分钟提醒你 / 未设置提醒 / 提醒时间已过，本次不会提醒"）区分精细，**没有用一句万能的"保存成功"糊弄**，这是内容设计上的认真。

## 八、可访问性与包容性

- 🟠 **P1 · 触控目标**：日程卡片的左侧时间锚点、历史页的密集文本块，需确认最小触控目标 ≥ 48dp，避免误触。卡片整体可点已达标，但卡片内若有可点子元素需注意重叠。
- 🟡 **P2 · 仅中文**：应用强制简体中文。作为课程作业无可厚非，但"强制"而非"跟随系统"对其他语言用户不友好。
- 🟡 **P2 · 无动态字号支持核查**：未确认文字是否随系统字号设置缩放；日程卡片在特大字号下可能溢出（时间锚点固定 minWidth、标题单行省略）。建议验证 `sp` 缩放下的布局稳健性。
- 🟡 **P2 · 色觉依赖**：提醒信息用青色区分，若仅靠颜色传达"有无提醒"则对色觉障碍用户不友好。当前有文字（"分钟前提醒"）兜底，问题不大，但需保持"信息不单靠颜色"的原则。

## 九、情感化设计

**"日程助手"的拟人化语感**是本应用的差异化机会，目前已显雏形：

- ✅ **今日摘要**："今天有 3 个日程 / 最近：15:00 组会 · 图书馆"——这一句让看板从"冷列表"变成"有个助手在跟你汇报"，方向正确。
- ✅ **保存反馈**："将在开始前 15 分钟提醒你"——第二人称、口语化，比"保存成功"有温度。

**尚可深化的情感化点**：

- 🟡 **P2 · 输入框可更"对话感"**：当前输入框是一个孤立的 EditText。若改成对话气泡样式（"跟我说说你有什么安排…"），会更强化"助手"人格，呼应产品概念。
- 🟡 **P2 · 空态可更有人情**：今日无日程时，"今天还没有日程"可以更温暖——"今天清空了，享受空闲吧 😊"或类似，避免工具感过冷（需拿捏，过度卖萌反而减分）。
- 🟡 **P2 · 缺少成就/连续反馈**：连续记录日程、本周已完成 N 项等轻量化激励，能提升长期留存，但属锦上添花。

## 十一、设计亮点（值得保持）

1. **永不死胡同的产品韧性**——AI 非必需、手动始终在，是成熟的产品判断。
2. **草稿→确认的人机分工**——享受 AI 省力，守住用户掌控。
3. **端到端的视觉一致性**——毛玻璃语言贯彻到自研选择器，无风格断层。
4. **已成型的设计系统**——令牌化的圆角/间距/尺寸，告别"每个卡片不一样"。
5. **时间锚点卡片**——信息优先级（时间>标题>地点>提醒）符合真实扫读习惯。
6. **今日摘要 + 保存反馈**——从"冷工具"走向"有语感的助手"，方向正确。
7. **昼夜双适配**——兼顾不同环境可读性。
8. **浅层级导航**——最多 3 层，回退清晰。

## 十二、总体评价

这是一个**概念锋利、视觉用心、韧性到位**的应用。它在"差异化概念（一句话建日程）""视觉识别度（毛玻璃）""产品韧性（AI 可降级）"三件最难的事上都交出了合格以上的答卷，设计系统的建立让它从"调界面"跨入了"有体系"。

当前最需要补的不是更多功能，而是**把已有的核心闭环做扎实**——两个 P0（删除无确认、澄清问题丢失）都是主流程上的硬伤，修复成本极低、收益极高；几个 P1（完成状态、时长、权限时机、降级延续）决定了它从"能演示"走向"能日常用"。视觉与情感化方向已经走在正确的路上，继续深化"助手人格"将是它真正的护城河。

**一句话总评**：设计语言已成体系，产品概念足够锋利；现在缺的不是花，而是把主流程的几处断裂缝补上，让"一句话到提醒"这条主干真正顺滑、可信。
